# 减法重构执行计划

本计划把项目收敛到一条清晰的骨架上，解决"功能界限不明确、模块联动混乱、维护成本高"的问题。**只做减法和收边界，不做大而全。**

## 已确认的方向决策

| 决策 | 内容 |
|---|---|
| 产品主场景 | 画布是产品，其他一切是画布的配件；取舍只问"是否让画布上迭代图片更顺" |
| 工作台 | 生图/视频独立工作台**并入画布**，不再单开页面；生成能力只在画布节点上操作 |
| 模块结构 | 导航收敛为：**画布 / 智能体 / 灵感广场 / 我的 / 配置**。智能体=工作流（按 图片/视频/文档 子分类）；灵感广场=只读公共内容源（内置精选+画廊，按重点关键词分类）；**我的**=个人空间（收藏 / 我的提示词 / 素材 / 生成记录） |
| 广场与画布的联动 | 广场条目只有**复制提示词**和**收藏**两个按钮；收藏落入"我的→收藏"；用户到画布中手动粘贴提示词，不做深度联动（画布内提示词选择弹窗一并移除） |
| BizyAir | 整条线删除（已停止国内运营），任务中心机制剥离泛化为统一任务运行时 |
| Provider | 配置页推荐 东木-AI（聚合自描述）+ RunningHub（云工作流），裸 OpenAI/Gemini 兼容渠道收入"高级"作保底；架构不锁死 |
| 媒体存储 | COS 是媒体唯一真源（输入直链 + 产物转存），IndexedDB 降级为元数据 + 离线缓存；WebDAV 四域同步整条移除 |
| 智能体来源 | 三条：RunningHub 工作流登记、**画布连线保存为智能体模板**（本计划内执行）、**AI 角色迁为"文档"子分类智能体**（本计划内执行） |
| 数据兼容 | 按 AGENTS.md：未上线，不写旧数据迁移和兼容分支，存储结构直接按新设计改 |

### 参考：东木-AI 的产品形态（3dmds.com）

从其前端可确认的结构，作为本计划的形态参照：画布即工作流（"用节点连线串联工作流，一键运行从想法到成品"、"运行画布"按钮）；智能体=可复用的模板；灵感广场=作品+提示词集市；收藏=「把当前作品收纳到你的私人精选库中，便于跨智能体复用」。**我们采其信息架构，不采其集市化**（发布/定价/分成不做）。

## 目标架构

```
配置层   Provider 注册表：东木(聚合·自描述) ｜ RunningHub(云工作流) ｜ 兼容渠道(保底)
执行层   统一任务运行时：submit → taskId → 轮询 → 终态 → 结果转存 COS → 回写画布
资产层   COS = 媒体唯一真源；IndexedDB = 元数据 + 离线缓存
界面层   画布(唯一创作场所，节点=生成，连线=工作流雏形)
         + 智能体(RunningHub登记 ｜ 画布模板 ｜ 文档角色)
         + 灵感广场(只读内容源：复制/收藏)
         + 我的(收藏 ｜ 我的提示词 ｜ 素材 ｜ 生成记录)
```

模块间联动收敛为三条单向通道：灵感广场 →(收藏)→ 我的；灵感广场/我的 →(复制)→ 用户 →(粘贴)→ 画布；画布 →(生成产物)→ 我的·素材/COS。

## 阶段划分

依赖顺序：`0 → 1 → 2`，然后 `3 / 4` 可并行，`5 / 6` 可并行收尾（6 中的画布模板依赖 2 的运行时）。

- `web/public/config.json`：删除 WebDAV 账号密码（随阶段 3 整体删 webdav 键）和"阿拉神灯"渠道的真实 apiKey，示例渠道 apiKey 留空。
- 文档注明：历史提交中的密钥视为已泄露，换新 COS 账号时直接作废旧密钥；新密钥用 CAM 子账号，只授予单桶单前缀的 PutObject/GetObject/DeleteObject 最小权限。

**验收**：全库 grep 无真实密钥；配置页手填 COS 后上传队列工作正常。

**删除清单**：
- `services/api/bizyair-workflows.ts`（唯一调用方是 runner）
- `components/canvas/canvas-node-upscale-dialog.tsx`、`canvas-drawing-render-dialog.tsx`（纯 BizyAir 功能）
- `multi-angle-camera-preview.tsx` 及 `canvas-node-angle-dialog.tsx` 中的 "BizyAir 双相机" 分支（保留"通用模型"模式）
- `pages/canvas/project.tsx` 三个 handler（约 2073 超分 / 2104 图纸渲染 / 2209 多角度）及其配置检查跳转
- `app-config-modal.tsx` "专业工作流" tab、`use-config-store` 的 `workflowConfig.bizyair` 与 `updateBizyAirConfig`、`ConfigTabKey` 的 `workflows` 成员（阶段 2 会以 Provider 配置重建此 tab）

**保留并泛化**：
- `stores/use-workflow-task-store.ts` 原样保留（任务运行时核心）
- `hooks/use-workflow-task-runner.ts` 抽掉 BizyAir 调用，保留通用骨架：任务循环、AbortController Map、5s 轮询、30min 超时、跨路由存活；提交/轮询动作改为按 `task.provider` 分发的接口（阶段 2 由 Provider 实现填充）

- `pages/canvas/project.tsx` 约 561-600 行的结果回写 effect 保留（provider 无关，只认 `task.resultUrls`）
- `lib/workflow-task-image.ts` 原样保留

**验收**：全库 grep 无 "bizyair"（不区分大小写）；画布图片生成任务、任务中心徽标/取消/重试/定位功能正常；多角度对话框仅剩通用模式且可用。

### 阶段 2：Provider 抽象 + 东木/RunningHub 接入

探查发现的现状问题：提供商身份靠 URL 后缀嗅探（`buildApiUrl` 的 `/v1`、`/api/v3`、`/api/plan/v3`）和模型名嗅探（`isSeedanceVideoConfig`）推断；`ApiCallFormat` 是 `'openai'|'gemini'` 闭合联合；能力路由靠模型名子串猜测；视频轮询循环在 `services/api/video.ts` 和 `pages/video/index.tsx` 各写一份。

**定义 Provider 接口**（`web/src/services/providers/`）：
- 身份与配置：provider 类型从渠道记录声明，彻底取代 URL/模型名嗅探
- 能力发现：`listCapabilities()` 返回模型/工作流清单 + 每项的参数 schema（select/textarea/number/upload/switch）+ 能力标签（text/image/video/audio、是否支持蒙版/参考图上限等）
- 文本流式：`streamText()`（现散落在 `services/api/image.ts` 的 ~500 行 OpenAI Response API 与 Gemini SSE 解析器归位到兼容渠道 Provider 内）
- 媒体任务：`submit() → taskId`、`poll() → {状态, 进度, 终态?, 结果URL[]}`、`cancel()`，接入阶段 1 的统一运行时

**三个实现**：
1. **东木-AI**：`/v1/skills/models` 能力发现（含参数定义元数据），`/v1/media/generate` + `/v1/skills/task-status` 走统一运行时；chat 模型按 `api_format` 分发到对应格式。参考 `.claude/skills/ai-api-skill/API_CAPABILITIES.md`。
2. **RunningHub**：能力=用户登记的工作流（workflowId + 参数映射表），`/task/openapi/upload`(multipart) + `/task/openapi/create`(nodeInfoList) + `/task/openapi/outputs` 轮询（code 813 排队/804 运行/805 失败/0 完成）。前置：工作流须在平台手动跑通过一次。
3. **兼容渠道**（保底）：现有 openai/gemini 双格式逻辑收拢为一个 Provider 实现，Seedance/火山方舟归入其中。

**配置与画布侧 UI**：
- 配置页推荐东木 + RunningHub（各填一个 Key 即用），兼容渠道收进"高级"折叠
- 画布节点的设置浮层（image/video/audio popover）改为参数 schema 驱动渲染；兼容渠道暂保留现有手写面板，避免一次改动过大
- `use-config-store` 渠道结构按新设计直接改（不写迁移），`channelId::model` 编码与名字子串能力分类由 Provider 能力声明取代
- 生成入口只保留画布（工作台并入见阶段 4），画布生成调度统一走任务运行时

**验收**：东木文生图/文生视频在画布节点上跑通（任务中心可见进度、结果回写节点）；RunningHub 一个真实工作流跑通；兼容渠道旧路径生图正常；文本问答/AI 增强/角色对话正常。

### 阶段 3：存储收敛——COS 唯一媒体真源，移除 WebDAV

探查结论：WebDAV 同步唯一入口是配置弹窗的手动按钮（`syncAppDataToWebdav` 仅一个调用点），删除干净；COS 上传队列（本地优先、3 次重试退避、刷新恢复、同步中心抽屉）已是成熟骨架，方向不变。

**删除清单**：
- `services/webdav-sync.ts`、`services/app-sync.ts`（含四域 manifest 合并逻辑）
- `app-config-modal.tsx` webdav tab、`use-config-store` 的 `WebdavSyncConfig`、`web/public/config.json` 的 `webdav` 键
- README/features 文档中 WebDAV 相关描述

**强化 COS 为唯一真源**：
- 输出侧：任务成功后产物从平台 URL（东木 `result_url` / RunningHub `fileUrl`，均可能过期）下载→`uploadImage`/`uploadMediaFile` 落本地→自动入 COS 队列（现有机制），素材/节点记录本地 storageKey，COS 地址仍由 `use-cos-upload-store` 唯一映射
- 输入侧：给平台喂参考图统一走 `ensureCosMediaUrl`（`services/api/video.ts` 已用，推广到图片参考、RunningHub 文件输入——优先直链，替代其平台上传接口）
- 挂账（todo）：元数据（画布 JSON/提示词/生成记录）定期打包备份到同一 COS 桶；当前版本 IndexedDB 即本地真源

**验收**：全库 grep 无 "webdav"；生成→产物入 COS→断网刷新后本地可见、联网后队列续传；跨设备通过 COS 直链可访问媒体。

**删除清单**：

- 两页私有的日志/会话体系（`GenerationSettings`/`LogPanel`/`LogCard`/`buildLog`/`serializeLog` 等重复代码）及 localforage 中 `image_generation_logs`、`video_generation_logs` 两个存储域
- `workflow-task-drawer.tsx` 中对 `video_generation_logs` 的只读摘要区
- `pages/video` 里与 `services/api/video.ts` 重复的轮询循环（其职责已并入阶段 2 的统一任务运行时）

**保留/迁移**：
- `services/api/{image,video,audio}.ts` 服务层保留（画布同样在用），随阶段 2 收拢进 Provider 实现
- `image/video/audio-settings-panel.tsx` 保留（画布三个设置浮层在用；`video-settings-panel` 导出的 normalize 函数随工作台删除去掉外部消费者）
- 工作台特有而画布暂缺的小能力（如剪贴板批量导入参考图），确认画布对应节点面板已覆盖后再删；未覆盖的在画布补齐后删

**验收**：导航无生图/视频工作台入口；原工作台的所有生成场景（文生图、图生图、参考图、视频、TTS）在画布节点上均可完成；`bun run typecheck` 通过。

### 阶段 5：灵感广场 + "我的"个人空间

现有 `/prompts` 页与内嵌"灵感画廊"弹窗、`/assets` 素材页，重组为两个页面：**灵感广场**（只读浏览）+ **我的**（个人空间）。

**灵感广场**（`plaza`，slug 避开现有路径段）：
- 内容 = 内置精选 + 1446 条 NanoBanana 画廊，统一卡片流；按重点关键词分类（沿用画廊 category 字段 + 内置条目 tags/group 归并）
- 条目上只有两个动作按钮：**复制提示词**、**收藏**（收藏落入"我的→收藏"）
- 数据层复用：`services/api/gallery.ts`（模块级缓存、~2.5MB 按需加载）

**我的**（`me`，由 `pages/assets` 改造，`navigation-tools.ts` 的 `assets 我的素材` 条目改为 `me 我的`）：
- **收藏**：从广场收藏的条目（`usePromptStore.importPrompts` 入本地库并标记收藏来源），可取消收藏
- **我的提示词**：用户自建/编辑的提示词（含"AI 增强"），新建/编辑/分组管理入口在此（原 prompts 页的 CRUD 与编辑器组件迁入）

- **生成记录**：`use-workflow-task-store` 任务历史视图（缩略图、状态、耗时、定位到画布节点、失败重试），复用 `workflow-task-drawer` 的展示逻辑

**联动收敛（重要减法）**：
- 删除画布内提示词选择弹窗链路：`components/canvas/canvas-prompt-library.tsx`、`components/prompts/prompt-select-dialog.tsx` 及其在画布节点面板的挂载点（工作台挂载点随阶段 4 已删）
- 删除"保存提示词为素材"（`savePromptAsset`）链路——提示词归"我的→提示词"，素材只放媒体
- 提示词到画布唯一路径 = 复制 → 手动粘贴

**必须保持**：
- `GALLERY_GROUP='灵感精选'` 常量与 `scripts/build-gallery.mjs` 同步；画廊条目 id 复用为提示词 id（"已收藏"去重依赖）
- CC BY 4.0 © MeiGen.ai 署名脚注

**验收**：导航为 画布/智能体/灵感广场/我的/配置；广场搜索/关键词分类正常、复制→画布粘贴顺畅；收藏在"我的→收藏"可见可管理、去重正确；"我的"四个分区各自功能完整；首屏包体积无明显增长。

### 阶段 6：智能体模块（三条来源全部落地）

导航新增 **智能体** 入口，按 **图片 / 视频 / 文档** 三个子分类展示智能体卡片。

**类型定义**（`types/workflow.ts`）：
`智能体 = { id, name, category(图片|视频|文档), steps }`；`步骤 = { provider, capabilityRef(模型名或workflowId), 参数映射表(含提示词), 输入输出类型 }`。运行 = 依序把步骤交给统一任务运行时，产物回写画布。

**来源一：RunningHub 工作流登记**
- 登记表单：workflowId + 参数映射（哪些 nodeId/fieldName 暴露为用户输入）+ 归类子分类
- 登记即成可运行卡片，执行走阶段 2 的 RunningHub Provider

**来源二：画布连线保存为智能体模板**（东木"运行画布"形态）
- 数据基础已具备：`CanvasNodeData.metadata` 含 model/prompt/references 等全部生成参数，`CanvasConnection` 提供 from→to 边
- 保存：选中一组节点 → 按连线拓扑排序 → 生成节点（generationMode 为 image/video/audio/text 者）转为步骤，连线决定步骤间的产物传递；无上游连线的图片/文本节点声明为模板输入；提示词进入步骤参数（可在运行前编辑）
- 运行：在画布上实例化模板（重建节点+连线，输入槽由用户填充），逐步骤经统一任务运行时执行并回写对应节点
- 入口：画布多选节点后右键/工具栏"保存为智能体"；智能体页卡片"在画布中运行"

**来源三：AI 角色迁为"文档"子分类智能体**
- `AiRole {id,name,systemPrompt,avatar,description}` 映射为单步骤文档智能体：步骤=文本分析（systemPrompt + 选中节点内容 → 文本节点产物），即原 `runRoleWorkflow` 流程的模板化
- `public/roles.json` 内置角色转为内置文档智能体模板；`use-role-store` 的用户自建/编辑/删除角色数据按新结构直接改（不写迁移，未上线）
- 画布"专业角色"入口与角色管理弹窗改为从智能体（文档分类）读取；`metadata.roleId` 字段改为 `agentTemplateId`，旧字段直接废弃
- 角色对话抽屉（`canvas-role-chat-panel`）与 `chatHistories`：**删除**——对话能力由画布助手/全站 Agent 面板承担，文档智能体只保留"分析→产出文本节点"这一职责（单一职责，减法）

## 风险与注意

- `isGenerationCanceled` 等逻辑对中文错误文案做字符串匹配（如"请求已取消"），Provider 实现必须保持既有错误文案，否则取消/重试交互断裂。
- `use-workflow-task-store` 上限 100 条任务且 `clearCompletedTasks` 会清成功任务——作为"生成记录"数据源时，需把上限与清理策略重新评估（如上限提高、清理只清失败/取消），并确保缩略图走 `resultStorageKeys` 本地解析而非易过期的平台 URL。

- 画布是删完工作台后唯一的生成入口，阶段 4 验收必须逐场景过一遍（文生图/图生图/参考图/局部重绘/视频/TTS），不能只看 typecheck。
