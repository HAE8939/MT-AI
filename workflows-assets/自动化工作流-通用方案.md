# AI 图像自动化工作流 · 通用方案（平台版）

> **适用范围**：本项目全部 22 个室内设计图像工作流（场景加模特、指定材质替换、室内氛围转换、光影重塑、高清放大、多视角分镜等，全量清单见第十一节）。
> **配套资产**：`workflows/` 目录下 22 个工作流配置 JSON（已全部完成并校验）；`ima/` 目录下对应 UI 原型图。
>
> **本文档怎么读**：
> - 想理解这套方案**为什么有价值** → 读第一、二节
> - 想知道**具体怎么用**（用户操作 / 开发接入 / 效果调优）→ 读第三节
> - 要**动手实现后端** → 读第四~九节
> - 要**新增或打磨工作流** → 读第十节 SOP + 第十一节矩阵
> - 排期与避坑 → 读第十二、十三节

## 一、这套方案是什么：意义与价值

### 1.1 解决的核心问题：提示词鸿沟

AI 图像模型（gpt-image-1 等）的效果上限很高，但它只听得懂**高质量的英文指令式提示词**——要写清位置、材质、光照方向、透视关系、锁定范围，还要规避十几种典型翻车模式（悬浮、无接触阴影、纹理尺度失真、蒙版边缘断层……）。

而我们的用户是**室内设计师和普通业主**，他们只会说：

> "把这面墙换成微水泥" / "帮我把这个毛坯房装成奶油风" / "开一下这盏灯"

用户的一句大白话和模型需要的百词英文工程化提示词之间，隔着一道鸿沟。**这套方案的本质，就是用"LLM + 领域知识库"自动填平这道鸿沟**：

```
用户一句话（中文、口语、信息残缺）
        │
        ▼
  提示词引擎（GPT-5.5 级 LLM + 该玩法的 JSON 知识库）
  · 看图判断场景/光照/风格
  · 按分层公式扩写，补全用户没说的信息
  · 注入防翻车短语，锁定不该动的部分
        │
        ▼
高质量英文指令提示词（60~140词，直接喂给图像模型）
        │
        ▼
   gpt-image-1 出图 → 用户拿到结果
```

用户始终只需要"上传图 + 涂一下 + 说一句话"，工程复杂度全部被平台吃掉。

### 1.2 三大设计价值

**① 知识资产化——把"会写提示词"变成公司资产而非个人手艺**

每个玩法的提示词经验（什么公式好用、哪里容易翻车、什么词汇能出效果）全部沉淀在对应的 JSON 知识库里。它带来三个直接好处：

- **不依赖人**：调优提示词不需要懂代码，运营/设计师改 JSON 即可；
- **可持续进化**：每次翻车案例都能回流为 JSON 里的一条 `failureModes`，系统越用越强；
- **可复制**：未来拓展到建筑、电商、服装等新行业，复制的是"方法论 + JSON 格式"，不是重写系统。

**② 配置驱动——新增玩法零代码**

22 个工作流共用**同一条流水线、同一套前端组件、同一个后端**。玩法之间的全部差异（要不要蒙版、几张参考图、扩写规则、调哪个端点、出几张图）都写在各自的 JSON 里。

**新增一个玩法 = 往 `workflows/` 目录放一个 JSON 文件**——后端启动时自动扫描注册，前端入口自动出现，代码零改动。这意味着产品迭代速度只取决于"写 JSON 的速度"，而不是开发排期。

**③ 可观测可调优——效果不靠玄学**

每次生成都把 LLM 扩写出的最终提示词（`final_prompt`）回传保存。出图不理想时，能明确区分是哪一环的问题：

| 现象 | 问题定位 | 修复动作 |
|---|---|---|
| final_prompt 就写得不好 | JSON 知识库缺口 | 补模板/词汇/规则 |
| final_prompt 很好但图不对 | 图像模型层问题 | 补 failureModes 防错短语、调参数 |
| 用户输入就有歧义 | 交互引导问题 | 改前端提示文案 |

### 1.3 一句话总结

> **这是一个"室内设计 AI 玩法工厂"：一条通用流水线 + 22 个（可无限扩展的）JSON 知识库插件。卖给用户的是"一句话出专业效果图"，沉淀给自己的是一套可复用、可进化、可复制的提示词工程资产。**

## 二、一次生成到底发生了什么（端到端实例）

以「局部材质修改」为例，完整走一遍数据流。**看懂这个例子，就看懂了整套系统。**

**【用户操作】**（全程 <30 秒）
1. 进入「局部材质修改」，上传一张客厅照片
2. 在蒙版区把沙发背景墙涂抹覆盖
3. 材料下拉框选「微水泥」，补一句"颜色浅一点"
4. 点「立即生成」→ 界面进入等待态

**【后端第 1 步：预处理】**
- 校验图片格式/大小；蒙版与原图分辨率对齐
- **蒙版 alpha 转换**：用户涂抹处 → 透明（alpha=0），未涂处 → 不透明。这是 gpt-image-1 的硬性要求：透明区=允许重绘区（全部 masked-edit 类工作流的公共步骤，只在预处理层实现一次）

**【后端第 2 步：提示词扩写】**
组装 LLM 请求：
- **System Prompt** = 固定引导语 + `局部材质修改.json` 的 `promptEngine` 全文（llmGuidance 双层效力声明、coreFormula 分层公式、materialLibrary 词汇库、failureModes 防错短语……）
- **User 消息** = 用户输入（"微水泥，颜色浅一点"）+ 原图（让 LLM 看图判断场景与光照）+ 蒙版语义（可选：后端预分析出"蒙版覆盖沙发背景墙"）

LLM（GPT-5.5）按知识库扩写，产出**最终英文提示词**，例如：

> *Change the sofa feature wall in the masked area to seamless microcement in a light warm grey with a subtle trowel texture and matte finish. Keep the wall's geometry and outline unchanged, preserving the existing artwork shadows and the wall-washing light gradient on the new surface. The trowel texture stays fine and understated. Do not alter anything outside the edited area.*

注意 LLM 自动完成了：中文材料名→完整英文材质描述（颜色+纹理+饰面）、"浅一点"→light warm grey、根据看图补充了挂画阴影与洗墙光的保留声明、结尾注入越界保护——这些用户全都没说，全部来自知识库。

**【后端第 3 步：调用图像模型】**
按该 JSON 的 `meta`：`POST /v1/images/edits`，携带原图 + 转换后的蒙版 + final_prompt。

**【后端第 4 步：返回】**
生成耗时 30~60s，全程异步（提交即返回 `task_id`，前端轮询/SSE）。完成后返回：结果图 + `final_prompt`（前端折叠展示，供调试回溯）。

**【用户视角】**：涂了一下墙、选了个材料 → 一分钟后拿到墙面换成浅色微水泥、光影完全自然的效果图。

## 三、使用方法：三种角色怎么用这套系统

### 3.1 终端用户（设计师/业主）怎么用

每个工作流的操作被统一约束为最多 4 步（各工作流按 `inputSpec` 自动增减）：

```
① 上传原图 → ②（部分玩法）涂抹蒙版 / 上传参考图 → ③（部分玩法）输入一句话或选选项 → ④ 点击生成
```

**用户侧使用要点**（前端应做成引导文案）：

| 场景 | 要点 |
|---|---|
| 蒙版类玩法 | 涂抹范围=修改范围。替换物体要连同其影子一起涂；「局部开灯」要把灯具+想被照亮的范围一起涂（光效出不了蒙版） |
| 参考图类玩法 | 材质参考用平整均匀的小样图；人物参考用正脸清晰照；风格参考用整屋实景而非单品特写 |
| 文字输入 | 一句话说清"哪里+改成什么"即可，不用写长描述——扩写是系统的事 |
| 多步链路 | 按推荐链路串联使用效果最佳（见 3.4） |

### 3.2 开发者怎么接入/实现

后端只需实现**一次**通用流水线，五个阶段全部由所选工作流的 JSON 配置驱动：

1. **启动注册**：扫描 `workflows/*.json` → 校验（`json.load` + 必填字段）→ 构建 Workflow Registry
2. **暴露接口**：`GET /api/workflows` 返回全部 `meta + inputSpec`，前端据此动态渲染入口列表与操作页（一套组件复用所有玩法）
3. **接收请求**：`POST /api/generate`（workflow_id + 各项输入）→ 按该工作流 `inputSpec` 校验 → 立即返回 `task_id`
4. **异步执行**：预处理（蒙版 alpha 转换等）→ 组装 System Prompt（见第七节）调 LLM 扩写 → 按 `meta.endpoint` 调图像模型 → 按 `outputSpec` 处理结果
5. **结果查询**：`GET /api/task/{task_id}` 返回状态 + 结果 + `final_prompt`

**开发者需要特别注意的三件事**：
- 蒙版 alpha 转换只写在预处理层一处（第十三节风险 1）；
- `final_prompt` 必须持久化并回传——这是整个调优体系的观测窗口，不是可选项；
- `hd-upscale` 是唯一不走 gpt-image-1 的工作流（`taskType: upscale`），路由到专业放大服务（Real-ESRGAN/Topaz/Magnific 类），其 JSON 的 `promptEngine` 仅在引导型放大服务时使用。

### 3.3 运营/调优者怎么用（无需写代码）

这套系统的效果上限由 JSON 知识库决定，而 JSON 是给"懂设计、懂效果"的人改的：

- **调效果**：某玩法出图不稳 → 查该玩法近期的 `final_prompt` 记录 → 判定缺口 → 改对应 JSON（补模板、补词汇、补防错短语）→ 保存即生效（无需发版）
- **新增玩法**：按第十节 SOP 六步走，全程不碰后端代码
- **改文案/选项**：前端下拉框选项直接来自 JSON（如 materialLibrary），加一种新材料=在 JSON 里加一行

**改 JSON 的边界**（对应 llmGuidance 双层效力设计，详见第六节）：
- 可自由改：templates、各词汇库、tips、buildExamples、failureModes 的增补——这些是"脚手架"，越丰富越好
- 谨慎改：outputContract、maskProtocol/referenceProtocol——这些是"硬契约"，约束的是图像模型特性，改错会系统性翻车

### 3.4 组合使用：推荐串联链路

单个工作流解决单点问题，**串联链路解决业务场景**（前端应做成"下一步推荐"引导）：

| 链路 | 串联 | 业务场景 |
|---|---|---|
| 洗图三部曲 | 光影重塑（去灰提亮）→ 质感增强（真实感）→ 高清放大（分辨率） | 渲染图/废片 → 可交付大图 |
| 出图链路 | 白膜出图 / 毛坯房出图 / 手绘线稿出图 → 洗图三部曲 | 从模型/毛坯/草图到成品效果图 |
| 平面链路 | 家装平面方案生成（结构图→布置图）→ 一键彩平图（布置图→彩平） | 户型图到客户能看懂的彩平 |
| 软装闭环 | 软装物料清单整理（实景→清单板）↔ 软装拼贴出图（拼贴→实景） | 方案沟通与采购物料双向转换 |
| 选区闭环 | 一键材质通道图（选区依据）→ 指定材质替换 / 局部材质修改 | 专业用户的精准局部修改 |

## 四、设计理念：一条流水线，工作流即插件

所有工作流共用**同一条通用流水线**：

```
上传输入 → （可选）蒙版/参考图 → 用户输入 → LLM 提示词引擎 → 图像模型 → 结果返回
```

各工作流之间的差异（要不要蒙版、扩写规则、调用哪个端点、输出几张图）**全部沉淀在各自的 JSON 配置文件里**，后端代码只实现一次通用逻辑。

**新增一个工作流 = 新增一个 JSON 配置文件，零代码改动。**

## 五、总体架构

```
┌───────────────── 前端（Web）─────────────────┐
│  工作流入口列表（由 /api/workflows 动态渲染）   │
│  ┌─────────────────────────────────────┐    │
│  │ 通用操作页：按 inputSpec 动态显示——     │    │
│  │ 上传区 / 蒙版涂抹区 / 参考图区 / 输入框  │    │
│  └─────────────────────────────────────┘    │
└───────────────────┬─────────────────────────┘
                    │ POST /api/generate（workflow_id + 各项输入）
                    ▼
┌───────────────── 后端（FastAPI）──────────────┐
│  Workflow Registry：启动时扫描 workflows/ 目录  │
│  加载全部 JSON 配置                            │
│      │                                       │
│      ▼ 按所选工作流的配置执行：                  │
│  ① 输入校验与预处理（尺寸对齐、蒙版alpha转换）    │
│  ② 提示词引擎：LLM + 该工作流的 JSON 知识库      │
│  ③ 调用配置指定的图像模型端点                    │
│  ④ 按 outputSpec 处理结果（单图/多图/文本）      │
│  ⑤ 任务状态管理与存储                           │
└───────────────────┬─────────────────────────┘
                    │ task_id → 前端轮询/SSE
                    ▼
               用户看到结果
```

## 六、通用流水线（五个阶段，按配置开关）

| 阶段 | 内容 | 由配置决定的部分 |
|---|---|---|
| 1. 输入采集 | 前端按 `inputSpec` 动态渲染操作区 | 是否需要蒙版、参考图数量、文字输入是否必填 |
| 2. 预处理 | 格式/大小校验；蒙版与原图尺寸对齐；**蒙版 alpha 转换**（涂抹处→透明=重绘区） | 是否有蒙版环节 |
| 3. 提示词扩写 | System = 该工作流 JSON 全文；User = 用户输入 +（推荐）原图，LLM 按 `coreFormula` 与 `outputContract` 产出最终英文提示词 | 扩写公式、模板库、约束规则——全部来自 JSON |
| 4. 模型调用 | 按 `meta.endpoint` 调用（带蒙版编辑 / 整图编辑 / 生成 / 放大） | 模型、端点、size/quality/n 等参数 |
| 5. 结果返回 | 按 `outputSpec` 处理：单图 / 多图组 / 结构化文本；存储并返回 `final_prompt` 供调试 | 输出类型与数量 |

> 通用规则：生成耗时 30~60s+，全链路异步（提交即返回 `task_id`）；`final_prompt` 永远回传前端可折叠查看，这是持续调优各工作流 JSON 的观测窗口。

### 工作流配置规范（JSON Schema）

每个工作流一个 JSON 文件，放在 `workflows/` 目录，结构分三层——**meta 管路由，inputSpec/outputSpec 管交互，promptEngine 管效果**：

```jsonc
{
  "meta": {
    "id": "scene-add-person",          // 唯一ID，前后端路由用
    "version": "2.2",
    "name": "场景加模特",               // 前端入口显示名
    "description": "……",
    "taskType": "masked-edit",         // masked-edit | full-edit | generate | upscale | multi-output | vision-analysis
    "targetModel": "gpt-image-1",
    "endpoint": "/v1/images/edits"
  },

  "inputSpec": {
    "image": "required",               // required | optional | none
    "mask": "required",                // required | optional | none
    "refImages": 0,                    // 参考图数量（如指定人物生成需要人物参考图）
    "userText": "required",            // 提示词输入
    "extraFields": []                  // 预设选项等，如材料下拉框、强度档位、分辨率选择
  },

  "outputSpec": {
    "type": "image",                   // image | images | text | file
    "count": 1                         // 多视角分镜等 > 1
  },

  "promptEngine": {                    // 提示词引擎知识库（各工作流的灵魂，差异最大的部分）
    "llmGuidance":   { },              // 双层效力声明：硬契约 vs 脚手架（见下方说明）
    "coreFormula":   { },              // 该玩法的扩写公式（分层解剖）
    "maskProtocol":  { },              // 有蒙版的工作流必填：蒙版为权威位置
    "referenceProtocol": { },          // 有参考图的工作流必填：图序约定 + 特征提炼规则
    "outputContract":{ },              // 英文/单段/长度上限/指令句式【硬契约】
    "intentFramework":{ },             // 保留层 / 引入层 / 协调层
    "templates":     [ ],              // 质量标杆 few-shot，每条 en+zh【脚手架】
    "attributes":    { },              // 属性词汇池（开放示例）【脚手架】
    "failureModes":  [ ],              // 常见翻车点 + 预防短语【硬契约（语义级）】
    "tips":          { },
    "buildExamples": [ ]               // rawInput → expanded_en 完整示例，≥2组
  }
}
```

> **llmGuidance 双层效力设计**（针对 GPT-5.5 级扩写模型）：知识库对 LLM 的约束分两级——
> - **硬契约**（严格遵守）：outputContract、maskProtocol/referenceProtocol 的权威关系、failureModes 的防错语义、用户指定优先。这些约束的是 gpt-image-1 的特性，与 LLM 智力无关，模型再强也不能松。
> - **脚手架**（质量标杆，鼓励超越）：templates/buildExamples 是 few-shot 示例而非套用模板；各词汇库（materialLibrary/styleLibrary 等）是开放示例，库外内容按同等信息密度自行展开；selectionRules/coherenceRules 收敛为原则，不机械执行。
>
> 设计理由：强模型吃「原则+示例」优于「规则+枚举」，封闭枚举会把表现拉平到模板水平。配合该设计，`final_prompt` 回传观测为必须项（自由度放开后输出方差变大，需可观测以区分知识库问题与模型发挥问题）。
>
> **编写规范**（所有工作流 JSON 必须遵守）：文件从 `{` 开始的合法 JSON；`outputContract` 必填；带蒙版的必写 `maskProtocol`；带参考图的必写 `referenceProtocol`；必含通用兜底模板；`buildExamples` 每组必含 `expanded_en`。

## 七、提示词引擎（通用机制）

```
System Prompt（按工作流动态组装）：
  你是{meta.name}方向的提示词专家。以下是本工作流的提示词知识库：
  {该工作流 JSON 的 promptEngine 全文}

  规则：
  1. 先阅读 llmGuidance，区分硬契约与脚手架
  2. 按 coreFormula 分层组织扩写，脚手架部分鼓励基于图与意图自由发挥
  3. 按当前场景判断风险，将相关 failureModes 防错语义融入提示词
  4. 严格遵守 outputContract，只输出最终提示词

User：
  用户输入：「{用户原文}」
  [有图时] 输入图：<image>   ← 推荐：让 LLM 看图判断场景/光照/风格
  [有参考图时] 参考图：<image>（图序按 referenceProtocol 约定）
  [有蒙版时] 蒙版语义：{后端预分析的蒙版位置描述，可选}
```

要点：

- **带图扩写为默认**：扩写 LLM 必须具备视觉能力（GPT-5.5 / Claude），看图判断场景类型、光照方向、色调，扩写质量显著优于纯文字。
- **图序即协议**：多图工作流严格按 referenceProtocol 的图序传图（第一张=原图/几何权威，第二张=参考图/特征来源），LLM 先把参考图"翻译"成文字特征再写入提示词。
- **蒙版语义转换是所有 masked-edit 类工作流的公共易错点**：gpt-image-1 要求透明（alpha=0）区域=重绘区，后端统一做"涂抹处→透明"转换，蒙版分辨率必须与原图一致。
- **指令句式而非逗号堆砌**：gpt-image-1 编辑端点对自然语言指令句（Add… Match… Preserve…）响应更好。

## 八、接口设计

| 接口 | 方法 | 说明 |
|---|---|---|
| `/api/workflows` | GET | 返回全部工作流的 `meta` + `inputSpec`，前端据此渲染入口列表和操作页 |
| `/api/generate` | POST (multipart) | `workflow_id` + `image?` + `mask?` + `ref_images?` + `prompt?` + `extra?`；返回 `{task_id}` |
| `/api/task/{task_id}` | GET | `{status: pending/processing/done/failed, results: [...], final_prompt, error}` |

## 九、技术选型

| 层 | 选型 | 理由 |
|---|---|---|
| 前端 | Vue3 / React + Canvas（fabric.js） | 操作页按 `inputSpec` 动态渲染，一套组件复用所有工作流 |
| 后端 | FastAPI（Python） | 异步友好；启动时扫描 `workflows/` 目录注册 |
| 任务队列 | MVP：`BackgroundTasks` + 内存字典；上量后 Celery + Redis | 先跑通 |
| 扩写 LLM | GPT-5.5（需视觉能力；备选 Claude 系列） | 带图扩写；配合 llmGuidance 双层效力设计 |
| 生图 | OpenAI `gpt-image-1`（edits/generations） | 按各工作流 `meta` 配置 |
| 高清放大 | 专业放大服务（Real-ESRGAN 自部署 / Topaz / Magnific 类） | 生成式模型会重绘内容且分辨率受限，放大必须走专用通道 |
| 存储 | MVP：本地目录；上线：OSS / S3 | — |

**验证路径**：每个新工作流先用 Gradio 原型验证其 JSON 扩写质量，达标后无需改后端——把 JSON 放进 `workflows/` 即上线。

## 十、新增一个工作流的标准步骤（SOP）

1. **准备素材**：该玩法的 UI 原型图 + 收集 5~10 个效果好的提示词范例
2. **编写 JSON**：按第六节规范编写（重点打磨 `coreFormula`、`templates`、`failureModes`；llmGuidance 可直接复用现有文件中的通用版本）
3. **离线验证**：拿 3~5 组典型用户输入，人工跑「LLM 扩写 → 生图」，检查扩写质量与成图效果
4. **迭代模板**：根据翻车案例补充 `failureModes` 预防短语与词汇库
5. **放入 `workflows/` 目录**：后端自动注册，前端自动出现入口
6. **上线观测**：通过 `final_prompt` 回看扩写质量，持续回流优化 JSON

## 十三、风险与注意事项

1. **蒙版 alpha 转换**（masked-edit 类公共坑）：涂抹区→透明，尺寸必须与原图一致，在通用预处理层统一实现、只写一次。
2. **生成耗时长**：全链路必须异步，前端要有明确的进度/等待态；多图类工作流（分镜等）耗时更长，考虑分张返回。
3. **API 成本**：每次生成 = 1 次 LLM + ≥1 次图像模型调用；多图类成本翻倍；对外开放需限流或积分计费。
4. **内容安全**：gpt-image-1 自带审核，被拒时把失败原因友好反馈并允许改词重试。
5. **配置质量即产品质量**：平台化后代码不再是瓶颈，各工作流 JSON 的打磨（模板覆盖度、failureModes 完备性）决定效果上限——`final_prompt` 观测与回流机制必须从第一天就有。
6. **JSON 文件规范**：所有配置文件必须是从 `{` 开始的合法 JSON（避免复制粘贴混入杂质文本），入库前用 `json.load` 校验。
7. **高清放大特殊路由**：`hd-upscale` 不走 gpt-image-1（会重绘内容且分辨率上限约 1536px），必须路由到专业放大服务；这是通用流水线里唯一的模型分叉点，按 `taskType: upscale` 分流。
